# Story 13.2: Documentation Gate for Verification

Status: verifying

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a developer,
I want verification to require user-facing documentation,
So that the verifier can install and use the project from docs alone.

## Acceptance Criteria

1. **Given** the developer runs `codeharness verify --story <id>`, **when** `README.md` does not exist in the project root, **then** verification fails with exit code 1 and the specific error message `[FAIL] No README.md found — verification requires user documentation`. (AC:1) <!-- verification: cli-verifiable -->

2. **Given** the developer runs `codeharness init` in a project without `README.md`, **when** init completes, **then** it scaffolds a minimal `README.md` containing: project name (from `package.json` name or directory basename), installation command (`npm install -g codeharness`), basic usage section, and a CLI command reference auto-generated from Commander.js `--help` output. (AC:2) <!-- verification: cli-verifiable -->

3. **Given** the README generated by `codeharness init`, **when** a user reads it, **then** it includes a "Quick Start" section with the minimum commands to install, configure, and run the project (at minimum: install command, first-run command, and one example usage). (AC:3) <!-- verification: cli-verifiable -->

4. **Given** a verification Docker image has been built, **when** the developer runs `codeharness verify-env check`, **then** it validates that the CLI installed inside the Docker container actually works by running `docker exec codeharness-verify codeharness --help` (or equivalent) and checking exit code 0. (AC:4) <!-- verification: integration-required -->

5. **Given** the CLI doesn't work inside the container (install failed, missing deps, broken binary), **when** `codeharness verify-env check` runs, **then** the check fails with a clear error indicating a build/packaging bug — the built artifact is not functional. (AC:5) <!-- verification: integration-required -->

## Dev Notes

### Architecture Constraints

- **CLI orchestrates all verification** (Architecture Decision 8). The README check is a precondition in the CLI's verify pipeline — not something the agent checks independently.
- **All templates are TypeScript string literals** (Architecture Decision 6). The README template lives in `src/templates/readme.ts`, not as a standalone markdown file.
- **Documentation is load-bearing** (Architecture Decision 10). Wrong docs = verification failure = real bug. This story makes that architectural principle enforceable.
- **FR85 traceability**: This story directly implements FR85 ("Verification requires README.md with working installation instructions — missing or broken docs fail verification").

- `existsSync()` from `node:fs` — README.md existence check
- `execFileSync` from `node:child_process` — capture `codeharness --help` output for CLI reference

### Key Design Decisions

1. **README check goes before preconditions** in `verifyStory()`. Without docs, verification is meaningless — the verifier needs docs to know how to use the software.
2. **README is NOT overwritten if it already exists.** Same pattern as AGENTS.md in init — idempotent, no data loss.
3. **CLI Reference is auto-generated** from Commander.js help output. This ensures docs stay in sync with actual CLI commands. If a command is added/removed, `codeharness init` regenerates the reference.
4. **verify-env check's `cliWorks` is the same check** described in AC4/AC5. Story 13.1 already implemented this. This story ensures the error messaging explicitly calls out build/packaging bugs.

### Anti-Patterns to Avoid

- Do NOT hardcode CLI commands in the README template — generate from Commander.js help output.
- Do NOT check for README.md in `checkPreconditions()` — it's a separate gate that fires before preconditions (tests/coverage don't matter if there's no documentation).
- Do NOT create a README.md during `codeharness verify` — init creates docs, verify only validates they exist.
- Do NOT overwrite an existing README.md during init — user's README is authoritative.
