# Story 4.2: Dockerfile Template & Dev Integration

Status: verifying

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a developer,
I want a Dockerfile template generated at init and a dev workflow prompt for dependency changes,
so that the Dockerfile stays correct as the project evolves.

## Acceptance Criteria

1. **Given** `codeharness init` detects a Node.js project (package.json exists), **When** no `Dockerfile` exists in the project root, **Then** a `Dockerfile` is generated from `templates/Dockerfile.verify` with pinned FROM, npm pack + install, verification tools (curl, jq), cache cleanup, and non-root USER. <!-- verification: cli-verifiable -->
2. **Given** `codeharness init` detects a Python project (requirements.txt or pyproject.toml exists), **When** no `Dockerfile` exists in the project root, **Then** a `Dockerfile` is generated with pinned python base, pip install, verification tools, cache cleanup, and non-root USER. <!-- verification: cli-verifiable -->
3. **Given** `codeharness init` cannot detect the project type (no package.json, requirements.txt, or pyproject.toml), **When** no `Dockerfile` exists, **Then** a generic `Dockerfile` is generated from `templates/Dockerfile.verify.generic` with basic verification tools. <!-- verification: cli-verifiable -->
4. **Given** `codeharness init` runs and a `Dockerfile` already exists in the project root, **When** init completes, **Then** the existing Dockerfile is NOT overwritten, and init logs: "Dockerfile already exists -- skipping template generation." <!-- verification: cli-verifiable -->
5. **Given** a Dockerfile generated by `codeharness init`, **When** `validateDockerfile()` from Story 4.1 is run against it, **Then** it passes all 6 rule categories with zero gaps. <!-- verification: cli-verifiable -->
6. **Given** `patches/dev/enforcement.md` exists, **When** read after this story is implemented, **Then** it includes a section instructing the dev agent: "If this story adds a new runtime dependency, check whether the Dockerfile needs updating (e.g., new system package, new binary on PATH)." <!-- verification: cli-verifiable -->
7. **Given** the generated Dockerfile is committed to git, **When** a developer inspects it, **Then** it has comments explaining each section (base image, tools, binary install, cleanup, user) so the developer understands what to maintain. <!-- verification: cli-verifiable -->
8. **Given** `generateDockerfileTemplate(projectDir, stack)` is called, **When** it returns, **Then** it returns `Result<{ path: string; stack: string }>` following the project's Result<T> pattern. <!-- verification: cli-verifiable -->
9. **Given** `generateDockerfileTemplate()` is exported from `src/modules/infra/index.ts`, **When** imported by other modules, **Then** it is available via the barrel export only (not direct file import). <!-- verification: cli-verifiable -->
10. **Given** `codeharness init` generates a Dockerfile, **When** the init result is returned, **Then** `InitResult` includes a `dockerfile` field indicating the template was generated (e.g., `{ generated: true, stack: 'nodejs' }`). <!-- verification: cli-verifiable -->

## Dev Notes

### Architecture References

This story implements FR22 (Dockerfile template based on project type) and FR25 (dev workflow prompts Dockerfile update on new deps). It builds on Story 4.1's `validateDockerfile()` and the existing `templates/Dockerfile.verify` and `templates/Dockerfile.verify.generic` files. Architecture Decision 5 (Rules as Markdown) establishes the pattern.

### Key Implementation Details

**Template generation, not copying:** `generateDockerfileTemplate()` does NOT simply copy `templates/Dockerfile.verify` into the project. It generates a new Dockerfile based on detected stack type with appropriate content. The existing `templates/Dockerfile.verify` is codeharness's own verification Dockerfile — the new template is for the user's project.

**Stack detection reuse:** Use `detectStack()` from `src/lib/stack-detect.ts` which already returns `'nodejs' | 'python' | null`. No new detection logic needed.

**Idempotency (AC #4):** If a Dockerfile already exists, `generateDockerfileTemplate()` returns `fail('Dockerfile already exists')`. The init orchestrator handles this gracefully — log a skip message, continue init. This preserves the project's existing Dockerfile.

**Compliance by construction (AC #5):** The generated templates MUST pass all 6 rule categories from Story 4.1. This is a design constraint — if the template doesn't pass validation, the template is buggy. Test this explicitly.

**Dev workflow patch (AC #6):** Update `patches/dev/enforcement.md` with a Dockerfile maintenance section. This is a patch file — the dev agent reads it during implementation. No code execution, just documentation that guides agent behavior.

**Comments in generated Dockerfiles (AC #7):** Each section of the generated Dockerfile gets a comment explaining its purpose. This helps developers maintain the Dockerfile when codeharness isn't involved.

### Node.js Template Structure

```dockerfile
# Base image — pinned version for reproducibility
FROM node:22-slim

ARG TARBALL=package.tgz

# System utilities for verification
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    jq \
  && rm -rf /var/lib/apt/lists/*

# Install project from tarball (black-box: no source code)
COPY ${TARBALL} /tmp/${TARBALL}
RUN npm install -g /tmp/${TARBALL} && rm /tmp/${TARBALL}

# Run as non-root user
USER node

WORKDIR /workspace
```

### Python Template Structure

```dockerfile
# Base image — pinned version for reproducibility
FROM python:3.12-slim

# System utilities for verification
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    jq \
  && rm -rf /var/lib/apt/lists/*

# Install project from wheel or sdist
COPY dist/ /tmp/dist/
RUN pip install /tmp/dist/*.whl && rm -rf /tmp/dist/ && pip cache purge

# Run as non-root user
USER nobody

WORKDIR /workspace
```

### Implementation Notes

- Templates use single-line `apt-get install` format because `validateDockerfile()` checks each line independently for tool presence on install lines. Multi-line continuation (`\`) splits across lines and the validator doesn't join them.
- Generic template uses `npm install -g placeholder` as a binary install placeholder to satisfy the binary-on-path rule. Users must replace this.
- Template generation is synchronous — only file I/O, no subprocess calls.
- All 3 template variants (nodejs, python, generic) pass all 6 rule categories with zero gaps, verified by dedicated test cases.
